[CC] Learned rule — "Dependabot PR rebase cadence when CI-changing commit lands after PR open". Under Infrastructure & Operations in `~/.claude/rules/learned-rules.md`. Pattern: when a Dependabot PR has a FAILURE check from a date preceding a CI-fix commit on main, don't assume upstream incompatibility — diff the workflow history (`git log -- .github/workflows/ci.yml`) against the PR open time. If a CI-fixing commit landed AFTER the PR opened, fresh `@dependabot rebase` will almost always flip it green. Cheap check: `gh run view <failing-run-id> --log-failed` (or raw `curl` against `/repos/.../actions/jobs/{id}/logs` if `gh` auth is cold) to see the actual error; confirm it's CI-env drift not an actions@v6 behavior change. Also: if 2+ Dependabot PRs touch the same workflow file (e.g., checkout + setup-node), merging the first invalidates the second's base — rebase the second AGAIN before merging. Session sptk-visual-audit 2026-04-21 executed this full cycle cleanly after one adjustment for the shared-file case.

[CC] Learned rule — "`gh` CLI auth timeout fallback: `GH_TOKEN=$GH_PAT gh ...` from `~/secrets.env`". Under Infrastructure & Operations. When `gh auth status` starts hanging/timing out ("Timeout trying to log in to github.com account ...") despite `curl -sI https://api.github.com/` returning 200, don't burn retries on `gh` with its cached credential. Source `~/secrets.env` (we have `GH_PAT`), export `GH_TOKEN=$GH_PAT` on a per-call basis, done — the `GH_TOKEN` env takes precedence over `gh auth`'s stored token and bypasses the hang entirely. Tiny pattern but rescues auth-flaky sessions quickly. Also: `gh run view --log-failed` doesn't always work with the same auth surface; curl against `/repos/<owner>/<repo>/actions/jobs/{id}/logs` is a clean fallback that returns a plain-text log. Session sptk-visual-audit 2026-04-21.
